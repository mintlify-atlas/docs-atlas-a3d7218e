Interactive prototypes produced by Web Design Skill are not wireframes or rough sketches — they are high-fidelity, clickable implementations with realistic device bezels, complete interaction state coverage, and a built-in live parameter panel. The goal is for a prototype to feel indistinguishable from an actual shipping product during a review session.

## What interactive prototypes cover

A prototype artifact typically includes one or more of the following:

- **Clickable app mockups** — screens connected by real interactions, not static images linked together
- **User flow exploration** — onboarding sequences, checkout paths, settings panels, and other multi-step flows
- **UI state exploration** — seeing the same component in every condition it can appear in, from default to error
- **Design variant comparison** — multiple layout, visual, or interaction directions shown side by side or toggled via the Tweaks panel

## Core requirements

### No title screen

Prototypes center in the viewport or fill it with sensible margins immediately. There is no splash screen, loading animation, or cover slide between the user and the prototype. The product appears at once.

### Device frames for realism

Wrapping prototype screens in device bezels grounds them in real-world context and prevents reviewers from evaluating the design as abstract pixels.

### At least 3 variants via the Tweaks panel

Every prototype exposes at least three design variants — toggled from the Tweaks panel rather than as separate files. This keeps the deliverable self-contained and lets reviewers compare directions in the same session without file management.

### Full state coverage

Every interactive component must appear in all states it can occupy:

## The Tweaks panel system

The Tweaks panel is a floating control surface in the bottom-right corner. It is completely hidden when closed so the prototype looks finished during a presentation, and opens to expose live design parameters.

```jsx
const TweaksPanel = ({ config, onChange, visible }) => {
  if (!visible) return null;

  return (
    <div style={{
      position: 'fixed',
      bottom: 20, right: 20,
      width: 280,
      background: 'rgba(24, 24, 27, 0.95)',
      backdropFilter: 'blur(12px)',
      borderRadius: 12,
      padding: 16,
      color: '#fff',
      fontSize: 13,
      zIndex: 9999,
      boxShadow: '0 20px 60px rgba(0,0,0,0.3)',
      border: '1px solid rgba(255,255,255,0.1)'
    }}>
      <div style={{ fontWeight: 600, marginBottom: 12, fontSize: 14 }}>Tweaks</div>

      {Object.entries(config).map(([key, value]) => (
        <div key={key} style={{ marginBottom: 12 }}>
          <label style={{ display: 'block', marginBottom: 4, opacity: 0.7 }}>
            {key}
          </label>
          {typeof value === 'boolean' ? (
            <input type="checkbox" checked={value}
              onChange={e => onChange({ ...config, [key]: e.target.checked })} />
          ) : typeof value === 'number' ? (
            <input type="range" min="0" max="100" value={value} style={{ width: '100%' }}
              onChange={e => onChange({ ...config, [key]: Number(e.target.value) })} />
          ) : value.startsWith('#') ? (
            <input type="color" value={value}
              onChange={e => onChange({ ...config, [key]: e.target.value })} />
          ) : (
            <input type="text" value={value}
              onChange={e => onChange({ ...config, [key]: e.target.value })}
              style={{
                width: '100%',
                background: 'rgba(255,255,255,0.1)',
                border: '1px solid rgba(255,255,255,0.2)',
                borderRadius: 4, padding: '4px 8px', color: '#fff'
              }} />
          )}
        </div>
      ))}
    </div>
  );
};
```

Design guidelines for the panel:
- Title is always **"Tweaks"** — consistent label across all prototypes
- Completely hidden when closed; the design must look final without it
- In multi-variant prototypes, variants are exposed as dropdowns or toggles inside Tweaks rather than separate files
- Even when the user does not ask for tweaks, add 1–2 creative parameters by default

## The three hard React rules

These rules are non-negotiable in every React prototype. Violating them produces silent bugs that are difficult to trace.

<Steps>
  <Step title="Never use `const styles = {}`">
    Multiple component files sharing `styles` as a global variable name silently overwrite each other. Always namespace with the component name:

    ```jsx
    // Correct
    const terminalStyles = { container: { ... }, line: { ... } };
    const headerStyles = { wrap: { ... } };

    // Wrong — will collide across files
    const styles = { container: { ... } };
    ```

    Or use inline `style={{...}}` directly. The variable name `styles` is banned.
  </Step>
  <Step title="Export cross-file components via `Object.assign(window, {})`">
    Separate `<script type="text/babel">` blocks do not share scope — each is compiled independently. To make a component available to other files, explicitly attach it to `window` at the end of the defining file:

    ```jsx
    function Terminal() { /* ... */ }
    function Line() { /* ... */ }

    Object.assign(window, { Terminal, Line });
    ```
  </Step>
  <Step title="Never use `scrollIntoView`">
    In iframe-embedded preview environments, `scrollIntoView` disrupts outer-frame scrolling. Use direct scroll assignments instead:

    ```jsx
    // Correct
    element.scrollTop = targetOffset;
    window.scrollTo({ top: targetOffset, behavior: 'smooth' });

    // Wrong
    element.scrollIntoView({ behavior: 'smooth' });
    ```
  </Step>
</Steps>

## Variant exploration strategy

Variants are not about delivering the single best option — they are about exhausting the design space so the reviewer can identify which elements resonate and mix them freely.

The skill explores at least four dimensions:

<AccordionGroup>
  <Accordion title="Layout variants">
    How the content is organized spatially: split pane, card grid, list view, timeline, full-bleed, sidebar+content. Layout changes should be distinct — not subtle shuffles.
  </Accordion>
  <Accordion title="Visual variants">
    Color palette, typography pairing, texture, layering depth, border-radius strategy. A light minimal variant alongside a dark atmospheric one covers a wide range quickly.
  </Accordion>
  <Accordion title="Interaction variants">
    Motion style, feedback patterns (color shift vs. scale vs. shadow), navigation metaphor (tabs vs. drawer vs. inline expand). Interaction variants change how the product *feels* to use.
  </Accordion>
  <Accordion title="Creative variants">
    Convention-breaking approaches — an unconventional layout metaphor, a strong visual concept, a novel navigation pattern. The first variants should be safe; the last should push boundaries.
  </Accordion>
</AccordionGroup>

<Tip>
Start the first variants safely within the design system, then progressively push boundaries. Show the full spectrum from "safe and functional" to "ambitious and daring." Reviewers pick the elements that resonate most and can ask for a hybrid.
</Tip>
