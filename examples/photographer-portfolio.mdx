This demo shows what the skill produces when given almost no direction — a single sentence with no section requirements, no color guidance, and no content to work with. Rather than producing a generic portfolio template, the skill invents a complete visual identity from first principles and builds the page as if a real designer had spent a week on it.

## The prompt

<Note>
"Build a homepage for an independent photographer's portfolio."
</Note>

One sentence. No name, no style, no color, no structure. Every decision in the output — character, palette, typography, layout, animation — was generated by the skill working through its design system declaration workflow.

## What the skill created

<AccordionGroup>
  <Accordion title="Character: a fictional identity from scratch">
    Rather than building a generic "Photographer Name" placeholder page, the skill invented a complete fictional identity: Mira Høst, a Nordic photographer with a distinctive visual voice. The name, implied geography, and aesthetic sensibility are all consistent — they feel like a real person's portfolio rather than a template waiting to be filled in.

    This reflects the skill's content philosophy: placeholders are honest markers for missing real content, but when the brief calls for a complete artifact, invented content should be internally consistent and credible. A fictional identity with a coherent point of view is more useful as a design reference than a page full of `[PHOTOGRAPHER NAME]` brackets.
  </Accordion>

  <Accordion title="Color: two-tone discipline">
    The entire page uses exactly two colors: paper-warm light (`#f2efe8`) and ink-dark (`#161513`). No accent color. No third tone.

    This is a deliberate constraint — one of the hardest to maintain in practice, because the instinct when a design feels flat is to reach for a third color rather than solve the problem through composition and typography. The skill treats the two-tone limit as a feature: it forces every visual decision to work harder.

    The specific values matter. `#f2efe8` is not pure white — it has a warm paper quality that reads as analog, tactile, photographic. `#161513` is not pure black — it has a brown-ink warmth that avoids the cold, screen-native feel of `#000000`. Together they evoke darkroom prints and editorial photography books.

    See the [oklch color system guide](/principles/oklch-color-system) for how to derive restrained palettes with perceptual precision.
  </Accordion>

  <Accordion title="Typography: Instrument Serif with extensive italic usage">
    The primary display face is Instrument Serif — a contemporary editorial serif with a strong italic that carries genuine calligraphic character rather than a simple slant. The skill pairs it with Space Grotesk for UI elements: navigation, labels, captions, and metadata.

    The italic usage is extensive and intentional. Section titles, quotes, and key phrases are set in italic as a design choice, not as emphasis. This creates a typographic rhythm across the page that reads as editorial — the kind of decision a type-conscious art director makes when laying out a photography book.

    The type scale uses a 5× ratio between display headings and body text, producing the visual contrast that makes editorial layouts read as composed rather than assembled.
  </Accordion>

  <Accordion title="Layout: magazine-editorial structure">
    The page uses numbered sections (01, 02, 03...) as a structural device — a technique borrowed from editorial design that creates visual rhythm and helps the reader track their position without conventional navigation anchors.

    Within sections, the layout uses asymmetric grids and side rails. The main content column does not span the full page width; a narrow side rail carries labels, dates, and metadata in a smaller type size. This creates a layered reading experience: primary content for skimming, secondary content for those who read closely.

    Asymmetric grids resist the default CSS Grid behavior of equal columns. The effort required to implement them is part of why AI-generated pages rarely use them — and why they're a reliable signal of intentional design.
  </Accordion>

  <Accordion title="Motion: Ken Burns and film grain">
    The hero image uses a slow Ken Burns animation — a gradual zoom and pan across a 24-second cycle, implemented in pure CSS keyframes. At 24 seconds, the motion is subliminal: visible when you're looking for it, unobtrusive when you're not. This duration was chosen deliberately; shorter cycles read as restless, longer ones disappear entirely.

    A film-grain texture overlay is applied to the hero using CSS — a subtle noise pattern that adds an analog, photographic quality to what would otherwise be a flat digital image placeholder. This is a technique that costs almost nothing to implement but dramatically shifts the perceived quality of the output.

    Both effects are implemented without JavaScript animation libraries — pure CSS transitions and keyframes. See the [animations output type guide](/output-types/animations) for the skill's approach to choosing animation complexity.
  </Accordion>

  <Accordion title="Navigation: mix-blend-mode: difference">
    The site masthead uses `mix-blend-mode: difference` — a CSS blend mode that inverts the masthead's colors based on whatever is behind it. Over the dark hero section, the masthead appears light. Over light content sections, it appears dark. The transition between states is seamless and requires no JavaScript.

    This technique solves a genuine design problem: how to maintain readable navigation across a page that moves between light and dark sections. The conventional solution — duplicating the navigation in two color variants and toggling them — is fragile and requires careful scroll tracking. The blend mode solution is a single implementation that works everywhere.

    It's also a technique that most AI-generated pages never reach. It requires knowing that the property exists, understanding how it behaves across different backgrounds, and making the deliberate choice to use it. The skill encodes that knowledge.
  </Accordion>
</AccordionGroup>

## Why these choices work

The portfolio succeeds as a design artifact because its choices form a coherent system rather than a collection of independent decisions. The two-tone palette requires the typography to carry all visual hierarchy — which is why the type scale and italic usage are so deliberate. The editorial grid structure creates the depth that the limited color palette cannot. The Ken Burns animation and film grain add tactile quality that a static, flat page in these colors would lack.

Each choice enables or requires the others. This is what separates designed artifacts from assembled ones: in a designed artifact, you can't change one element without reconsidering the rest.

<Tip>
The skill surfaces these dependencies in its design system declaration step. When you read the declaration for a project like this one, you'll see how the two-tone constraint shapes every downstream decision — typography, layout, motion, detail treatment.
</Tip>

## How to prompt for similar results

The photographer portfolio demo was produced from a minimal prompt. That works because the skill fills the decision space rather than waiting to be told what to decide. But certain prompt patterns produce more consistent results.

**What works well**

Prompts that specify a subject and a mood without over-specifying visual decisions:

```text
Build a homepage for an independent photographer's portfolio.
```

```text
Create a portfolio site for a ceramic artist. Quiet, handmade feel.
```

```text
Design a landing page for a documentary filmmaker. Serious, editorial.
```

The mood adjective — "quiet," "editorial," "serious" — gives the skill enough signal to make coherent color and typography choices. Without it, the decisions are still deliberate, but mood alignment is more variable.

**What to avoid**

Specifying visual decisions that the skill should make:

```text
Build a portfolio with a black and white color scheme, Helvetica font, and a full-screen hero image.
```

This over-constrains the design system declaration step and produces worse outcomes than letting the skill derive the system from the mood and subject. If you have specific requirements, express them as constraints (`the palette should feel cold and austere`) rather than prescriptions (`use black and white`).

**What signals the skill is active**

Look for the design system declaration in the agent's response — the Markdown block that specifies color palette, typography, spacing system, border-radius strategy, shadow hierarchy, and motion style before any code is written. If you see this block, the skill's workflow is running. If the agent jumps straight to code, the skill may not have engaged.

See the [design system declaration guide](/workflow/design-system-declaration) for the full format.
